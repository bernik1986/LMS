# План разработки учебной платформы

Документ подготовлен на основе ТЗ: закрытая кастомная LMS-платформа с личным кабинетом студента, админ-панелью, ручным созданием пользователей, назначением курсов, контролем прогресса, тестированием и сертификатами.

## 1. Цель продукта

Создать закрытую онлайн-платформу для обучения, где:

- внешний пользователь может оставить заявку на курс;
- аккаунты создаются только администратором вручную;
- администратор управляет пользователями, курсами, материалами, тестами, назначениями и сертификатами;
- студент проходит только назначенные ему курсы;
- тест открывается только после прохождения обязательных материалов;
- после успешного теста курс считается завершенным, а студент получает сертификат.

## 2. Границы MVP

### Входит в первую версию

- Публичная форма заявки на курс.
- Админ-панель для просмотра заявок.
- Авторизация пользователей без самостоятельной регистрации.
- Роли `student` и `admin`.
- Ручное создание, редактирование, деактивация и удаление пользователей.
- Хранение персональных данных пользователя, включая фото.
- Создание и редактирование курсов.
- Создание уроков или разделов внутри курса.
- Загрузка и хранение учебных материалов: видео, аудио, текст, PDF, изображения, файлы для скачивания.
- Назначение одного или нескольких курсов пользователю.
- Отслеживание прогресса по обязательным материалам.
- Последовательное прохождение материалов, если это включено для курса.
- Открытие теста только после завершения обязательной учебной части.
- Конструктор теста для курса.
- Минимально обязательный тип вопроса: один правильный ответ из нескольких вариантов.
- Настройка количества попыток и проходного процента.
- Прохождение теста студентом.
- Сохранение истории попыток.
- Автоматическое определение результата: сдан / не сдан.
- Выдача сертификата после успешного завершения курса.
- Личный кабинет студента с курсами, прогрессом, тестами и сертификатами.
- Админский просмотр результатов, прогресса и сертификатов.
- Базовые e-mail уведомления.

### Лучше вынести во вторую версию

- Онлайн-оплата курсов.
- Личный кабинет компании.
- Массовое назначение курсов.
- Импорт пользователей из Excel.
- Экспорт результатов в Excel.
- Проверка сертификата по QR-коду.
- Многоязычность интерфейса.
- Внутренний чат с администратором.
- Автоматические напоминания о незавершенном курсе.
- Расширенная аналитика по компаниям и курсам.
- Текстовые ответы с ручной проверкой.
- Вопросы с несколькими правильными ответами, если сроки MVP ограничены.

## 3. Рекомендуемая архитектура

Архитектуру лучше строить модульно, чтобы админ-панель, студентский кабинет, тесты и сертификаты не смешивались в один трудно поддерживаемый блок.

### Вариант стека, если он еще не выбран

- Frontend: Next.js или React.
- Backend: Next.js API, NestJS, Django, Laravel или другой привычный для команды фреймворк.
- Database: PostgreSQL.
- File storage: S3-совместимое хранилище, локальное хранилище допустимо только для ранней разработки.
- Auth: серверные сессии или JWT с refresh-token, обязательная защита админ-маршрутов.
- E-mail: SMTP, SendGrid, Mailgun, Postmark или аналог.
- PDF/сертификаты: HTML-шаблон с генерацией PDF на сервере.

### Основные слои

- Публичная часть: страница курса или сайта, кнопка заявки, форма заявки.
- Студентский кабинет: профиль, курсы, прохождение материалов, тесты, сертификаты.
- Админ-панель: управление заявками, пользователями, курсами, назначениями, тестами, результатами и сертификатами.
- Backend API: бизнес-логика, права доступа, проверки прогресса, тесты, сертификаты, уведомления.
- Хранилище файлов: фото пользователей, материалы курсов, сертификаты.
- База данных: пользователи, курсы, прогресс, попытки, заявки, сертификаты.

## 4. Основные роли и права

### Студент

- Входит в личный кабинет.
- Видит только свои данные.
- Видит только назначенные ему курсы.
- Проходит доступные материалы.
- Получает доступ к тесту только после выполнения условий курса.
- Проходит тест в пределах разрешенного количества попыток.
- Видит результаты своих тестов.
- Скачивает свои сертификаты.

### Администратор

- Видит админ-панель.
- Управляет заявками.
- Создает и редактирует пользователей.
- Активирует и деактивирует пользователей.
- Сбрасывает пароли.
- Управляет курсами и материалами.
- Назначает курсы пользователям.
- Настраивает тесты, попытки и проходной процент.
- Видит прогресс и результаты студентов.
- Управляет сертификатами.

## 5. Основные пользовательские сценарии

### Сценарий 1. Заявка на курс

1. Пользователь нажимает кнопку "Зарегистрироваться на курс".
2. Открывается форма заявки.
3. Пользователь заполняет фамилию, имя, телефон, e-mail, выбирает курс и при необходимости оставляет комментарий.
4. Система валидирует поля.
5. Заявка сохраняется в админ-панели.
6. Администратору отправляется e-mail уведомление.
7. Пользователь видит экран успешной отправки.
8. Аккаунт автоматически не создается.

### Сценарий 2. Создание пользователя администратором

1. Администратор открывает заявку или раздел пользователей.
2. Создает пользователя вручную.
3. Заполняет персональные данные.
4. Загружает фото при необходимости.
5. Указывает e-mail и начальный пароль или запускает письмо для установки пароля.
6. Активирует пользователя.
7. Назначает один или несколько курсов.
8. Пользователь получает письмо с доступом.

### Сценарий 3. Прохождение курса студентом

1. Студент входит в личный кабинет.
2. Открывает раздел "Мои курсы".
3. Выбирает активный курс.
4. Проходит обязательные материалы.
5. Система фиксирует просмотр или завершение каждого материала.
6. Если курс требует последовательности, следующий материал открывается только после предыдущего.
7. После выполнения условий открывается тест.
8. Студент проходит тест.
9. Система считает процент правильных ответов.
10. Если проходной процент достигнут, курс завершается.
11. Система создает сертификат.

### Сценарий 4. Неуспешный тест

1. Студент проходит тест.
2. Система считает результат ниже проходного процента.
3. Студент видит процент результата и статус "не сдан".
4. Система показывает количество оставшихся попыток.
5. Если попытки остались, студент может повторить тест.
6. Если попытки закончились, повторное прохождение блокируется.
7. Администратор видит неуспешную попытку в результатах.

## 8. Статусы и бизнес-правила

### Пользователь

- `active`: может войти и проходить курсы.
- `inactive`: не может войти или получать доступ к курсам.

### Назначение курса

- `not_started`: курс назначен, но студент не открывал материалы.
- `in_progress`: студент начал прохождение.
- `materials_completed`: все обязательные материалы завершены.
- `test_available`: тест открыт.
- `test_failed`: последняя попытка неуспешна.
- `test_passed`: тест сдан.
- `completed`: курс завершен и сертификат выдан или готов к выдаче.

### Доступ к тесту

Тест доступен только если:

- пользователь активен;
- курс активен;
- курс назначен пользователю;
- назначение не удалено и не завершено с блокировкой;
- все обязательные материалы имеют статус `completed`;
- лимит попыток не исчерпан;
- тест активен.

### Завершение курса

Курс считается завершенным только если:

- обязательные материалы завершены;
- тест сдан с результатом не ниже проходного процента;
- создана запись сертификата или подтверждено, что сертификат не требуется для конкретного курса.

## 9. План разработки по этапам

### Этап 0. Уточнение требований и проектирование

Ориентир: 2-4 рабочих дня.

Задачи:

- Уточнить выбранный стек.
- Определить формат сертификата: PDF, HTML, изображение, ручная загрузка или автогенерация.
- Уточнить критерий просмотра видео: открытие, процент просмотра, просмотр до конца.
- Уточнить, где хранить файлы: локально, S3, облако.
- Уточнить e-mail провайдера.
- Уточнить дизайн: готовый UI kit, кастомный дизайн или быстрый административный интерфейс.
- Согласовать MVP и список функций второй версии.
- Подготовить прототипы основных экранов.

Результат:

- утвержденный состав MVP;
- схема ролей и прав;
- структура базы данных;
- список экранов;
- план релизов.

### Этап 1. Базовая инфраструктура проекта

Ориентир: 3-5 рабочих дней.

Задачи:

- Создать репозиторий и структуру проекта.
- Настроить frontend и backend.
- Подключить базу данных.
- Настроить миграции.
- Подключить линтеры, форматирование и базовые тесты.
- Настроить переменные окружения.
- Подготовить Docker или инструкции локального запуска.
- Добавить базовую систему файлового хранения.
- Подготовить seed-данные: первый администратор и тестовый курс.

Результат:

- проект запускается локально;
- есть база данных и миграции;
- можно создать первого администратора;
- есть основа для API и интерфейсов.

### Этап 2. Авторизация, роли и безопасность

Ориентир: 4-6 рабочих дней.

Задачи:

- Реализовать вход по e-mail и паролю.
- Реализовать выход из системы.
- Добавить защиту маршрутов.
- Добавить роли `admin` и `student`.
- Запретить публичную регистрацию.
- Добавить создание пароля администратором или письмо для установки пароля.
- Добавить сброс пароля.
- Защитить админ-панель от доступа студентов.
- Добавить middleware или guard для проверки прав.

Результат:

- администратор и студент входят в разные зоны;
- публичной регистрации нет;
- права доступа проверяются на backend, а не только в интерфейсе.

### Этап 3. Заявки на курс

Ориентир: 2-4 рабочих дня.

Задачи:

- Создать публичную форму заявки.
- Добавить поля из ТЗ.
- Подключить список активных курсов.
- Реализовать серверную валидацию.
- Сохранять заявку в базе.

- Реализовать загрузку файлов.
- Добавить флаг обязательности материала.
- Добавить флаг последовательного прохождения курса.
- Реализовать предпросмотр курса для администратора.

Результат:

- администратор может самостоятельно собрать курс;

- Реализовать последовательную разблокировку материалов.

- администратор видит прогресс.

### Этап 7. Конструктор тестов

Ориентир: 6-10 рабочих дней.

Задачи:

- Создать сущность теста для курса.
- Реализовать настройки теста: попытки, проходной процент, время, показ результата, повторное прохождение.
- Реализовать добавление вопросов.
- Реализовать редактирование и удаление вопросов.
- Реализовать варианты ответов.
- Реализовать выбор правильного ответа.
- На первом этапе поддержать `single_choice`.
- Добавить валидацию: у вопроса должен быть текст и минимум два варианта, один вариант должен быть правильным.
- Добавить предпросмотр теста для администратора.

Результат:

- администратор может собрать тест без разработчика;
- тест привязан к конкретному курсу.

### Этап 8. Прохождение тестов студентом

Ориентир: 5-8 рабочих дней.

Задачи:

- Проверять доступность теста перед стартом.
- Блокировать тест до завершения обязательных материалов.
- Создавать попытку теста.
- Показывать вопросы студенту.
- Принимать ответы.
- Считать результат.
- Сравнивать результат с проходным процентом.
- Учитывать лимит попыток.
- Показывать успешный или неуспешный результат.
- Сохранять историю попыток.
- Обновлять статус назначения курса.
- Показывать результаты в личном кабинете и админ-панели.

Результат:

- студент проходит тест в рамках правил курса;
- система корректно определяет результат;
- история попыток сохраняется.

### Этап 9. Сертификаты

Ориентир: 4-8 рабочих дней.

Задачи:

- Создать сущность сертификата.
- Генерировать уникальный номер сертификата.
- Создавать сертификат после успешного завершения курса.
- Сохранять снимок данных пользователя и курса на момент выдачи.
- Подготовить шаблон сертификата.
- Сформировать файл сертификата или страницу просмотра.
- Добавить скачивание сертификата студентом.
- Добавить просмотр и скачивание сертификата администратором.
- Добавить повторную отправку сертификата на e-mail.
- Добавить возможность ручной корректировки данных сертификата, если это нужно бизнесу.

Результат:

- сертификат связан с пользователем, курсом и назначением;
- студент видит сертификат в личном кабинете;
- администратор управляет выданными сертификатами.

### Этап 10. Уведомления

Ориентир: 3-5 рабочих дней.

Задачи:

- Настроить e-mail провайдера.
- Подготовить шаблоны писем.
- Отправлять администратору уведомление о новой заявке.
- Отправлять студенту уведомление о назначении курса.
- Отправлять студенту уведомление об успешном прохождении.
- Отправлять студенту уведомление о доступном сертификате.
- Опционально отправлять уведомление о неуспешной попытке теста.
- Логировать статусы отправки.

Результат:

- ключевые события сопровождаются e-mail уведомлениями;
- неудачные отправки можно отследить.

### Этап 11. Админский дашборд и ежедневная работа

Ориентир: 4-6 рабочих дней.

Задачи:

- Добавить краткие метрики: новые заявки, активные студенты, активные курсы, завершенные курсы.
- Добавить быстрые ссылки: создать пользователя, создать курс, посмотреть заявки, посмотреть результаты.
- Добавить фильтры и поиск в основных таблицах.
- Добавить понятные статусы и бейджи.
- Добавить пагинацию.
- Добавить действия из таблиц: назначить курс, сбросить пароль, открыть карточку.

Результат:

- админ-панель удобна для регулярной работы;
- основные действия выполняются быстро.

### Этап 12. Тестирование и приемка

Ориентир: 5-8 рабочих дней.

Задачи:

- Написать unit-тесты на бизнес-правила.
- Проверить права доступа.
- Проверить невозможность публичной регистрации.
- Проверить блокировку теста до прохождения материалов.
- Проверить лимиты попыток.
- Проверить расчет проходного процента.
- Проверить генерацию сертификата.
- Проверить e-mail уведомления.
- Провести ручное тестирование основных сценариев.
- Исправить найденные ошибки.
- Подготовить приемочный чек-лист.

Результат:

- MVP готов к демонстрации и приемке;
- критические бизнес-правила покрыты тестами.

### Этап 13. Деплой и запуск

Ориентир: 3-6 рабочих дней.

Задачи:

- Подготовить production окружение.
- Настроить базу данных.
- Настроить файловое хранилище.
- Настроить домен и HTTPS.
- Настроить резервные копии.
- Настроить логирование ошибок.
- Настроить мониторинг доступности.
- Создать первого администратора.
- Загрузить первые курсы.
- Провести финальное smoke-тестирование.

Результат:

- платформа доступна пользователям;
- есть базовая эксплуатационная защита: бэкапы, логи, мониторинг.

## 11. Критерии приемки MVP

- Внешний пользователь может отправить заявку на курс.
- После отправки заявки аккаунт не создается автоматически.
- Администратор видит заявку в админ-панели.
- Администратор может создать пользователя вручную.
- Студент может войти в личный кабинет.
- Неактивный студент не может пользоваться платформой.
- Администратор может создать курс с материалами.
- Администратор может назначить курс студенту.
- Студент видит назначенный курс.
- Студент не видит неназначенные курсы.
- Студент не может открыть тест до завершения обязательных материалов.
- После завершения обязательных материалов тест становится доступным.
- Студент может пройти тест.
- Система учитывает лимит попыток.
- Система рассчитывает процент правильных ответов.
- Если процент ниже проходного, курс не завершается.
- Если процент равен или выше проходного, курс завершается.
- После успешного теста создается сертификат.
- Сертификат доступен студенту в личном кабинете.
- Администратор видит прогресс, результаты и сертификаты пользователя.
- История прохождения и тестов сохраняется.

## 12. Риски и вопросы для уточнения

- Какой стек разработки предпочтителен?
- Нужен ли готовый дизайн или достаточно аккуратного административного интерфейса?
- Что считать просмотром видео: открытие, 80-90 процентов просмотра или просмотр до конца?
- Нужно ли защищать видео от скачивания?
- Где будут храниться файлы и какой ожидаемый объем видео?
- Нужна ли автогенерация PDF-сертификата в MVP или достаточно записи сертификата и страницы просмотра?
- Должен ли сертификат содержать фото пользователя всегда или только для отдельных курсов?
- Кто задает шаблон сертификата?
- Нужно ли администратору редактировать уже выданный сертификат или лучше делать перевыпуск?
- Нужна ли юридическая обработка персональных данных и согласие на нее в форме заявки?
- Нужны ли разные администраторы с ограниченными правами?
- Требуется ли журнал действий администратора для аудита?
- Какие e-mail уведомления обязательны для первого релиза?
- Нужно ли ограничение по сроку доступа к курсу?
- Нужно ли повторное прохождение курса после завершения?

## 13. Ориентировочная оценка сроков

Оценка зависит от команды, выбранного стека, дизайна и требований к сертификатам/видео.

### Быстрый MVP

Ориентир: 6-8 недель.

Подходит, если:

- команда использует знакомый стек;
- дизайн делается на базе готовой UI-библиотеки;
- поддерживается только один тип вопроса;
- сертификат простой;
- видео и файлы хранятся в стандартном хранилище без сложной защиты.

### Более полноценная версия

Ориентир: 9-12 недель.

Подходит, если:

- нужен более аккуратный UX;
- сертификаты должны генерироваться в PDF;
- нужна расширенная админ-панель;
- нужны подробные фильтры и отчеты;
- требуется стабильная production-инфраструктура.

## 15. Минимальный технический чек-лист перед запуском

- Все админские API защищены ролью администратора.
- Студентские API фильтруют данные по текущему пользователю.
- Нет публичного endpoint для самостоятельной регистрации.
- Пароли хранятся только в виде хеша.
- Файлы проверяются по типу и размеру.
- Фото и материалы недоступны без проверки прав, если они не публичные.
- Доступ к тесту проверяется на backend.
- Результат теста считается на backend.
- Попытки теста нельзя накрутить повторными запросами.
- Сертификат нельзя скачать чужому пользователю.
- Все критические действия администратора логируются.
- Есть резервные копии базы данных.
- Есть обработка ошибок отправки писем.
- Есть базовый мониторинг production-сервера.

## 16. Следующий практический шаг

Перед началом разработки стоит принять 5 решений:

1. Выбрать стек разработки.
2. Зафиксировать точный MVP.
3. Определить формат сертификата.
4. Определить способ хранения видео и файлов.
5. Подготовить прототипы основных экранов: админ-панель, кабинет студента, страница курса, тест, сертификат.

После этого план можно превратить в backlog: эпики, user stories, задачи на backend, frontend, дизайн, тестирование и деплой.
