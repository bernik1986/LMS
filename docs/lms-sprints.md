# Спринтовый план разработки LMS

План основан на документе [lms-development-plan.md](./lms-development-plan.md). Формат: 2-недельные спринты. Если команда небольшая или разработка идет одним разработчиком, спринты можно воспринимать как последовательные этапы поставки.

## Sprint 0. Подготовка и фиксация MVP

Цель: убрать неопределенность перед разработкой и зафиксировать технические решения.

Задачи:

- Утвердить стек: Next.js, TypeScript, Prisma, PostgreSQL для production, SQLite для локальной разработки.
- Зафиксировать MVP: заявки, пользователи, курсы, материалы, назначения, тесты, сертификаты, базовые уведомления.
- Принять решение по сертификатам: HTML/PDF-шаблон, уникальный номер, снимок данных на момент выдачи.
- Принять решение по материалам: локальное хранилище в dev, S3-совместимое хранилище в production.
- Зафиксировать критерий завершения материала: для MVP ручная кнопка "Отметить как пройдено", позже процент просмотра видео.
- Подготовить карту экранов и основные пользовательские сценарии.

Результат спринта:

- Согласованы границы MVP.
- Есть техническая дорожная карта.
- Есть список открытых вопросов для бизнеса.

Статус: частично выполнен. План и модель MVP уже подготовлены.

## Sprint 1. Технический фундамент и каркас приложения

Цель: запустить базовый проект, базу данных, роли, seed-данные и каркас интерфейса.

> **Важно:** Sprint 1 в этом документе описывает **Next.js-каркас** (`src/app/`). Рабочий MVP реализован отдельно в `scripts/lms-server.mjs` и закрывает Sprint 2–8 полностью. Не путайте mock-страницы Next.js с рабочим приложением.

Задачи:

- Инициализировать Next.js + TypeScript проект.
- Настроить базовые скрипты: `dev`, `build`, `lint`, `prisma`.
- Подключить Prisma.
- Создать схему данных для MVP.
- Настроить SQLite для локальной разработки.
- Добавить seed-данные: администратор, студент, демо-курс, уроки, тест.
- Создать базовый layout и навигацию.
- Создать публичную главную страницу.
- Создать страницу входа.
- Создать заготовку админ-панели.
- Создать заготовку кабинета студента.
- Добавить базовые server-side helpers для ролей и доступа.

Критерии приемки:

- Проект устанавливается и запускается локально.
- База данных создается миграцией.
- Seed создает первого администратора и демо-данные.
- Есть публичная страница, страница входа, админская зона и студентская зона.
- В коде нет публичной самостоятельной регистрации.

Статус: выполнен для Next.js-каркаса (mock UI). Рабочее приложение — в standalone-сервере, см. блок «Текущий фокус».

## Sprint 2. Авторизация, пользователи и заявки

Цель: сделать закрытый вход, ручное создание пользователей и публичные заявки на курс.

Задачи:

- Реализовать вход по e-mail и паролю.
- Реализовать выход.
- Защитить маршруты администратора и студента.
- Реализовать публичную форму заявки.
- Сохранять заявки в базе.
- Добавить список заявок в админ-панели.
- Добавить статусы заявки.
- Реализовать создание пользователя администратором.
- Реализовать редактирование пользователя.
- Реализовать активацию/деактивацию пользователя.
- Реализовать сброс пароля.

Критерии приемки:

- Пользователь не может зарегистрироваться самостоятельно.
- Заявка не создает аккаунт.
- Администратор видит заявки.
- Администратор вручную создает пользователя.
- Неактивный пользователь не получает доступ к кабинету.

## Sprint 3. Курсы, уроки и материалы

Цель: дать администратору возможность создавать учебный контент.

Задачи:

- Реализовать CRUD курсов.
- Реализовать разделы/уроки курса.
- Реализовать материалы уроков: текст, видео-ссылка, PDF/файл, изображение.
- Добавить сортировку уроков и материалов.
- Добавить обязательность материалов.
- Добавить флаг последовательного прохождения курса.
- Добавить предпросмотр курса.
- Добавить загрузку файлов в локальное хранилище.

Критерии приемки:

- Администратор может создать активный курс.
- Курс содержит уроки и материалы.
- Материалы можно менять и упорядочивать.
- Неактивный курс не предлагается в форме заявки.

## Sprint 4. Назначения и прогресс обучения

Цель: сделать прохождение назначенных курсов студентом.

Задачи:

- Реализовать назначение курса пользователю.
- Реализовать удаление назначения.
- Показать назначенные курсы в кабинете студента.
- Реализовать страницу курса для студента.
- Фиксировать открытие и завершение материалов.
- Рассчитывать прогресс.
- Блокировать следующие материалы при последовательном прохождении.
- Отображать прогресс администратору.

Критерии приемки:

- Студент видит только свои назначенные курсы.
- Прогресс сохраняется.
- Обязательные материалы влияют на доступность теста.
- Администратор видит статус прохождения.

## Sprint 5. Конструктор тестов

Цель: дать администратору гибкую настройку теста для курса.

Задачи:

- Реализовать тест как часть курса.
- Реализовать настройки: попытки, проходной процент, время, показ результата, повторное прохождение.
- Реализовать вопросы типа `single_choice`.
- Реализовать варианты ответов.
- Реализовать выбор правильного ответа.
- Добавить валидацию теста.
- Добавить предпросмотр теста.

Критерии приемки:

- У курса может быть тест.
- Администратор может менять вопросы и ответы.
- У каждого вопроса есть один правильный ответ.
- Нельзя активировать некорректный тест.

## Sprint 6. Прохождение тестов

Цель: реализовать прохождение теста студентом и расчет результата.

Задачи:

- Проверять доступность теста на backend.
- Создавать попытку теста.
- Принимать ответы.
- Считать процент правильных ответов.
- Учитывать лимит попыток.
- Показывать результат студенту.
- Сохранять историю попыток.
- Обновлять статус назначения курса.

Критерии приемки:

- Тест нельзя открыть до завершения обязательных материалов.
- Попытки нельзя превысить.
- Результат считается на сервере.
- История попыток отображается студенту и администратору.

## Sprint 7. Сертификаты

Цель: выдавать сертификаты после успешного завершения курса.

Задачи:

- Реализовать сущность сертификата.
- Генерировать номер сертификата.
- Создавать сертификат после успешного теста.
- Сохранять снимок данных пользователя и курса.
- Реализовать страницу просмотра сертификата.
- Реализовать скачивание сертификата.
- Добавить список сертификатов в личном кабинете.
- Добавить управление сертификатами в админ-панели.

Критерии приемки:

- Сертификат связан с пользователем, курсом и назначением.
- Студент видит свои сертификаты.
- Студент не может открыть чужой сертификат.
- Администратор видит все выданные сертификаты.

## Sprint 8. Уведомления и удобство админ-панели

Цель: закрыть ежедневные рабочие процессы администратора.

Задачи:

- Настроить e-mail провайдера.
- Добавить шаблоны писем.
- Отправлять уведомление о новой заявке.
- Отправлять уведомление о назначении курса.
- Отправлять уведомление об успешном завершении курса.
- Отправлять уведомление о доступном сертификате.
- Добавить админский дашборд с метриками.
- Добавить фильтры, поиск и пагинацию.

Критерии приемки:

- Ключевые события отправляют письма.
- Ошибки отправки логируются.
- Администратор быстро находит заявки, пользователей, курсы и результаты.

## Sprint 9. Тестирование, hardening и деплой

Цель: подготовить MVP к демонстрации и запуску.

Задачи:

- Покрыть тестами бизнес-правила доступа к тесту.
- Покрыть тестами расчет результата.
- Проверить права доступа.
- Проверить защиту файлов и сертификатов.
- Проверить основные сценарии вручную.
- Настроить production env.
- Подготовить инструкции запуска.
- Настроить логирование и резервные копии.
- Провести smoke-тест после деплоя.

Критерии приемки:

- MVP проходит приемочный чек-лист.
- Критические бизнес-правила защищены тестами.
- Есть инструкция локального и production запуска.
